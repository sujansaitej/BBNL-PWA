1. Project Overview & Tech Stack

What: A Progressive Web Application for BBNL (Bharat Broadband Network Limited) that serves as a CRM for employees/franchisees and a self-service portal for customers. Includes live TV streaming (275+ channels), payment processing, ticket management, customer registration with KYC, and service management.

9. Caching Architecture (L1 — L4)

Request │ ▼ L1: In-Memory (channelStore Map, logoCache Map) Speed: 0ms │ Capacity: unlimited │ Lifetime: tab session │ miss ▼ L2: IndexedDB (channelStore → iptv-store DB) Speed: 2-5ms │ Capacity: ~250MB │ Lifetime: persistent │ miss ▼ L3: Cache API / Workbox (Service Worker) Speed: 10-50ms │ Capacity: varies │ Lifetime: TTL-based │ miss ▼ L4: localStorage (lsCache → _c: prefixed keys) Speed: 1-10ms │ Capacity: 5MB │ Lifetime: TTL-based │ miss ▼ Network → Response stored back into L1 + L2 (or L3/L4)

lsCache.js — localStorage Cache

  • All keys prefixed with _c: namespace
  • Each entry: { d: data, t: timestamp }
  • lsGet(key, ttlMs) — returns data if fresh, null if expired
  • lsGetStale(key, ttlMs) — returns { data, fresh } for stale-while-revalidate
  • lsSet(key, data) — writes with auto-eviction on quota exceeded
  • Lazy cleanup on first access removes expired entries
  • Migration: cleans up old unprefixed keys from previous builds

channelStore.js — IPTV Data Cache

  • L1: In-memory Map — synchronous reads (0ms)
  • L2: IndexedDB iptv-store database — persistent across sessions
  • Hydration: L2 → L1 on import (runs once via _hydratePromise)
  • TTL: 2G=60min, 3G/4G=30min, Languages=2x TTL
  • Keys: channels_{mobile}_subs, languages_{mobile}, ads_{mobile}_fofi

logoCache.js — Image Cache

  • L1: Map of URL → blob:// object URL
  • L2: Workbox Cache API (CacheFirst strategy)
  • Concurrency: 2G=6, 3G=30, 4G=100 parallel fetches
  • Priority queue for language logos (bypass concurrency cap)
  • Failure handling: 2 retries, 60s cooldown, sentinel values (NOT_FOUND vs null)

10. IPTV / Live TV System

Data Flow

App Startup (main.jsx) │ ├── channelStore hydrates L1 from L2 (IndexedDB) └── iptvPrefetch checks staleness │ ├── If fresh: preload language logos only └── If stale: fetch languages + channels in parallel │ └── Store in L1 + L2 (channelStore) User opens Live TV (LiveTvPage.jsx) │ ├── Read from L1 (instant) or wait for L2 hydration ├── Render channel list with lazy-loaded logos ├── Language filter bar at top (horizontal scroll) ├── Ad banner injected every 10 channels └── Voice search (Hindi/English/Tamil/Telugu/Kannada) User taps channel → PlayerPage.jsx │ ├── Check channel.streamlink (embedded in channel data) ├── Check sessionStorage cache (25-min TTL, XOR-obfuscated) ├── Fallback: call /ftauserstream API └── Play via HLS.js with adaptive config

Voice Search (useVoiceSearch.js)

Supports multilingual voice recognition with number parsing:

  • Languages: English, Hindi, Tamil, Telugu, Kannada
  • Converts spoken numbers: "ek sau pachas" → 150, "two hundred" → 200
  • Mic permission requested explicitly before use

11. HLS Streaming & Player

Stream URL Flow

Channel object from API │ ├── Has .streamlink? → normalizeStreamUrl() │ │ │ ├── Indian user (low RTT): │ │ Direct HTTPS → livestream.bbnl.in │ │ │ └── International user (high RTT): │ ├── Relay configured? → https://relay-us.domain.com/stream/host/path │ └── No relay → /smartphone/crm/stream/host/path (server proxy) │ └── No .streamlink? → call /ftauserstream API → get tokenized URL

High-Latency Detection (isHighLatencyConnection)

  1. Check navigator.connection.rtt (Chrome/Edge) — if > 150ms → high latency
  2. Fallback: check timezone — non-South-Asian timezone → high latency
  3. South Asian timezones: Kolkata, Calcutta, Colombo, Dhaka, Kathmandu, Karachi, Thimphu, Rangoon, Bangkok, Jakarta

  • Fullscreen: CSS rotation (not Fullscreen API) to avoid exposing domain name

15. Performance Optimizations

Build Level (vite.config.js)

  • Code Splitting: Manual chunks for hls.js, leaflet, jspdf, swiper, framer-motion, vendor-react
  • Module Preload Filtering: Heavy chunks NOT preloaded (prevents 1.4MB unnecessary downloads)
  • Console Stripping: Production removes console.log/debug/info + debugger statements
  • Tree Shaking: Vite/esbuild dead code elimination

Runtime Level

  • React.memo: Applied to IPTV sub-components (AdBanner, LangCard, ChannelRow, ChannelCard)
  • content-visibility: auto on list children — skips layout/paint for off-screen rows
  • CSS containment: contain: layout style on list items
  • Lazy image loading: loading="lazy" + IntersectionObserver
  • Connection-aware tuning: Timeouts, concurrency, cache TTLs adapt to 2G/3G/4G
  • Navigation Preload: SW fires network request before boot completes (50-100ms savings)

16. Security & Logging

Security Features

  • Stream URL obfuscation: XOR cipher + base64 in sessionStorage (prevents plain-text URL exposure)
  • No Fullscreen API: Uses CSS rotation to avoid browser showing domain to user
  • CORS origin/referer gate: Stream proxy validates request origin before proxying
  • Host allowlist: Only STREAM_HOSTS hostnames can be proxied
  • Directory traversal prevention: server.js validates file paths stay within dist/
  • nginx security headers: HSTS, X-Content-Type-Options, X-Frame-Options, X-XSS-Protection

Structured Logging (utils/logger.js)

  • Dev: All levels (debug, info, warn, error, security)
  • Prod: warn, error, security only
  • Auto-redacts: password, Authorization, x-api-key, token, secret, credit_card, cookie
  • Security events: LOGIN_SUCCESS, LOGIN_REJECTED, OTP_VERIFY, LOGOUT, SESSION_EXPIRED, API_AUTH_REJECTED

19. International Streaming (Relay Server)

Problem

Users in USA/Canada/Australia experience 200-350ms RTT to Indian stream servers. Each HLS segment fetch crosses the ocean, causing buffering.

Solution: Edge Relay Servers

Without Relay: User (USA) ── 200ms ──→ livestream.bbnl.in (India) 200ms ←── = 400ms per segment With Relay: User (USA) ── 15ms ──→ relay-us (USA VPS) ── 200ms ──→ India (cached!) User (USA) ← 15ms ── relay-us = 30ms per segment (cache hit)

Deployment File

deployment-package/relay-server.js — standalone relay that only handles /stream/* requests with segment caching (200MB, 30s TTL).

Configuration

# In .env.production (after deploying relay VPS):
VITE_RELAY_US=https://relay-us.yourdomain.com
VITE_RELAY_AU=https://relay-au.yourdomain.com

How Routing Works

  1. PlayerPage.jsx calls isHighLatencyConnection()
  2. If high latency + relay configured: URL becomes https://relay-us.domain.com/stream/host/path
  3. If high latency + no relay: URL becomes /smartphone/crm/stream/host/path (main server proxy)
  4. If low latency (India): direct HTTPS to livestream.bbnl.in

23. Pending Client Issues / Bugs to Fix

The following bugs were reported by the client (via WhatsApp, March 2026). They are unresolved and need investigation by the next developer.

BUG-001 iPhone — Current Location Not Working
Platform: iOS (iPhone) Severity: High Page: Registration / Ticket Creation (map picker)

On iPhones, the "Use Current Location" button does not capture the user's GPS location. The app either silently fails or the location value stays empty. This affects registration flows and ticket creation where geolocation is required.

Investigation hints: Check navigator.geolocation.getCurrentPosition() calls across the app. iOS Safari requires the page to be served over HTTPS and needs explicit <meta name="apple-mobile-web-app-capable"> for PWA. Look at src/pages/Register.jsx and src/pages/TicketsMap.jsx — the Leaflet locate() call may need a longer timeout and enableHighAccuracy: true on iOS. Also check if the PWA manifest correctly requests geolocation permissions.
BUG-002 Android — Map Pick Not Taking Location Until You Drag the Map Icon
Platform: Android Severity: Medium Page: Registration / Ticket Creation (map picker)

On Android devices, after tapping the map to select a location, the selected coordinates are not registered until the user physically drags the map pin/marker. Simply tapping a location on the map should be enough, but the value only updates after a drag event.

Investigation hints: The map picker likely listens for dragend event on the marker but not click on the map. Check the Leaflet marker setup in src/pages/Register.jsx and src/pages/TicketsMap.jsx. Add a map.on('click', (e) => ...) handler that updates the coordinates immediately, in addition to the existing marker.on('dragend'). Also verify src/pages/SmartNavigationMap.jsx for similar patterns.
BUG-003 Filter Option Is Not Working
Platform: All Severity: Medium Page: Customer List / Tickets

The filter functionality on list pages (customer list, tickets list) does not produce correct results or has no effect when filter options are selected. The list data remains unchanged after applying filters.

Investigation hints: Check src/pages/Customerlist.jsx and src/pages/Tickets.jsx for filter state management. Look for useState / useEffect hooks that control filtering logic — the filter state may not be wired to the API call params or the local array .filter() predicate. Also verify that the API endpoints (src/services/customer/apis.js, src/services/generalApis.js) accept and respect filter query parameters.
BUG-004 After Payment — Shows "No Payment Data Available" Error
Platform: All Severity: High Page: Paynow / FoFi Payment

After completing a payment successfully, the app shows a "No payment data available" error message instead of the payment confirmation or receipt. The payment may have gone through on the backend, but the frontend fails to display the result.

Investigation hints: Check src/pages/Paynow.jsx and src/pages/FofiPayment.jsx. The issue is likely in the payment response handling — either the API response format changed and the code isn't parsing the new structure, or there's a race condition where the success redirect happens before the payment data is stored in state. Look at src/services/orderApis.js for the payment API response shape and compare it against what the component expects. Also check if the payment gateway callback URL is correctly routing back to the app.
BUG-005 FoFi SmartBox — Scan from TV Shows "Camera Access Required" Error
Platform: All (Mobile browsers) Severity: Medium Page: FoFi SmartBox Service → QR Scanner

When users attempt to scan a QR code from the TV screen using the FoFi SmartBox feature, the app shows a "Camera Access Required" error. The camera permission prompt may not appear, or the app fails to access the camera even when permission has been granted.

Investigation hints: Check src/pages/services/FoFiSmartBox.jsx — the QR scanning uses jsqr library with navigator.mediaDevices.getUserMedia(). Common issues: (1) The app must be on HTTPS for camera access, (2) iOS requires the playsinline attribute on the video element, (3) The camera stream may not be properly released after scanning causing subsequent attempts to fail — look for stream.getTracks().forEach(t => t.stop()) in cleanup. Also check if the permission check (navigator.permissions.query) is correctly implemented — it's not supported in all browsers and Safari handles it differently.
Priority order (suggested): BUG-004 (payment) → BUG-001 (iPhone location) → BUG-005 (camera) → BUG-002 (Android map) → BUG-003 (filters). Payment and location bugs directly impact core user flows.
